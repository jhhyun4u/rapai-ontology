## 1. 시스템 정의

### 1.1 한 줄 정의

RAPai(Research AI Platform)는 L1 TRL Gate·L2 Stage Gate·L3 Annual
Gate·L4 Moving Target Gate의 4계층 Gate 체계를 자동 운영하고, 프로젝트
유형(6종)과 TRL·CRL 단계를 동시에 인식하여 관리 프레임워크를 자동
전환하며, PMBOK 기반 10개 도메인 에이전트가 LangGraph StateGraph로
협업하는 R&D·학술용역 통합 프로젝트 관리 플랫폼입니다.

### 1.2 v2.5 설계 원칙

> **v2.5 핵심 설계 원칙 --- 4계층 Gate 체계 기반**
|                                                                       |
| 원칙 1 --- Gate는 4계층이다: L1(TRL 기술 확인)·L2(투자                |
| Go/Kill)·L3(정부R&D 연차·단계)·L4(Moving Target 유효성)는 서로 다른   |
| 목적·주체·주기를 가진 독립 계층이다.                                  |
|                                                                       |
| 원칙 2 --- TRL과 Gate는 다르다: TRL 달성(L1)은 기술적 사실 확인이며   |
| Go/Kill이 없다. Stage Gate(L2)는 경영적 투자 판단이며 TRL 달성이 L2의 |
| Must 기준 중 하나다.                                                  |
|                                                                       |
| 원칙 3 --- 정부R&D 연차평가(L3)는 L2와 독립: L2는 내부                |
| 의사결정(PI·PM), L3는 외부 행정 평가(전문기관·평가위원회).            |
| 주기·기준·결정 구조가 모두 다르다.                                    |
|                                                                       |
| 원칙 4 --- Gate 결정은 인간이 한다: RAPai는 패키지를 조립하고 근거를  |
| 제시할 뿐이다. 모든 Gate 최종 결정은 반드시 PI 서명을 요구한다.       |
|                                                                       |
| 원칙 5 --- 유형별 Gate 활성화: 유형1·2에는 L1\~L4 전부 활성화.        |
| 유형3·4·5에는 L2(품질/마일스톤 Gate)·L3(해당 시)만 활성화. L1·L4는    |
| 기술 R&D 전용.                                                        |

## 4. Gate 4계층 체계 --- 완전 정의

### 4.1 Gate 개념 정의 --- 왜 4계층인가

R&D 현장에서 \"게이트\"라는 말은 실제로 목적·주체·주기·결정구조가 전혀
다른 4가지 개념을 혼용합니다. RAPai는 이를 명확히 분리하여 4계층으로
정의합니다.

> **Gate의 본질적 정의 (Cooper Stage-Gate 원전 기반)**
|                                                                       |
| Gate는 단순한 진도 보고나 정기 회의가 아닙니다.                       |
|                                                                       |
| Gate는 \"이 프로젝트를 계속 진행할 것인가, 중단할 것인가, 방향을 바꿀 |
| 것인가\"를 결정하는 공식 의사결정 점입니다.                           |
|                                                                       |
| Gate는 반드시 세 가지 요소를 포함해야 합니다: ① 판단 기준(Criteria) ② |
| 결정(Decision) ③ 다음 단계 자원·계획 확약(Output).                    |
|                                                                       |
| 그러나 \"TRL 달성 확인\"과 \"투자 계속 판단\"과 \"정부 평가\"는       |
| 목적·주체·결정구조가 모두 달라 하나의 Gate로 처리할 수 없습니다.      |

### 4.2 L1 --- TRL Gate (기술 성숙도 확인 Gate)

  -----------------------------------------------------------------------
  **항목**          **내용**
  ----------------- -----------------------------------------------------
  정의              TRL이 목표 단계에 달성됐음을 PI가 근거와 함께
                    서명·확인하는 기술적 사실 확인 Gate.

  목적              기술 성숙도 증명. 다음 TRL 구간 관리 모드 전환 준비.

  적용 유형         유형1 원천기술·유형2 기술사업화 R&D만 적용.
                    유형3·4·5는 없음.

  발동 주기         비정기·이벤트 기반. TRL_ACHIEVED 이벤트 발생 시 자동
                    트리거.

  판단 기준         TRL 달성 근거 (ELN 기록·실험 데이터·논문 DOI·특허
                    번호·실증 보고서). CRL 현재값 확인.

  결정 구조         달성(통과) / 미달(추가 연구 필요). Go/Kill 없음 ---
                    Kill은 L2에서 결정.

  결정 주체         PI 서명 필수. 외부 전문가 검토 선택적.

  RAPai 처리        TRL_ACHIEVED 이벤트 → AG-KPI: TRL_SNAPSHOT
                    생성(Append-only, PI 서명 포함) → KPI_RECONFIG 이벤트
                    → AG-SCOPE: WBS 재편 초안 → AG-REPORT: TRL 전환
                    보고서.

  L2와의 관계       TRL Gate(L1) 통과 ≠ Stage Gate(L2) 통과. L1은 기술
                    사실 확인이고 L2는 투자 지속의 경영적 판단. TRL
                    달성이 L2 Must 기준 중 하나임.

## 20. Gate 거버넌스 --- 생성·유지·운영 체계

### 20.1 Gate 거버넌스 개요

Gate 체계를 \"설계\"하는 것과 실제로 \"운영\"하는 것은 다른 문제입니다.
이 장은 4계층 Gate를 누가 만들고(생성), 어떻게 유지하며(유지), 실제로
어떻게 운영하는가(운영)의 거버넌스 전체를 정의합니다.

> **Gate 거버넌스 3가지 핵심 원칙**
|                                                                       |
| 원칙 1 --- RAPai는 결정하지 않는다: RAPai는 패키지 조립·경보 발송·SLA |
| 모니터링·GATE_DECISION 기록만 담당. 모든 Gate 결정은 반드시           |
| 사람(PI·PM·기관관리자·외부 평가위원회)이 내린다.                      |
|                                                                       |
| 원칙 2 --- 기준은 버전 관리된다: 모든 Gate 기준 변경은 Append-only.   |
| 이전 버전 삭제·수정 불가. Gate 결정 시 \"어떤 버전 기준으로           |
| 판단했는가\"가 자동 기록된다.                                         |
|                                                                       |
| 원칙 3 --- 계층별 주체가 다르다: L1=PI, L2=PI+PM, L3=외부             |
| 평가위원회+기관관리자, L4=PI+기관관리자. 각 계층의 결정               |
| 주체·SLA·에스컬레이션 경로가 명확히 구분된다.                         |

### 23.6 Override·오작동 정정·책임 체계

> **책임 귀속 최종 원칙**
|                                                                       |

|                                                                       |
| 모든 중대 결정(Gate Go/Kill·Pivot·과제 중단)의 법적·행정적 책임은     |
| 결정 서명자(PI·관리담당자)에게 귀속된다.                              |
|                                                                       |

| 서명이 있는 경우 그 결정의 효력은 유지되며 별도 이의신청 절차를 통해  |
| 재심의한다.                                                           |
|                                                                       |
| Human은 언제든 AI 에이전트의 판단을 Override(재정의)할 수 있다.       |

## 부록 A. 주요 용어 정의 (v2.7 통합)

  -------------------------------------------------------------------------------------------------
  **용어**               **정의**
  ---------------------- --------------------------------------------------------------------------
  TRL (Technology        기술 성숙도 지수. TRL 1(기본 원리 관찰)\~9(실제 환경 검증 완료) 9단계. L1
  Readiness Level)       Gate의 핵심 지표. PI가 달성 근거와 함께 서명하여 확정. TRL_SNAPSHOT 노드로
                         영구 보존.

  CRL (Customer          고객 준비도 지수. CRL 1(시장 문제 인식)\~9(정식 계약·반복 구매) 9단계.
  Readiness Level)       TRL과 함께 L2 Gate Must 기준을 구성. CRL 5(고객 PoC 완료)가 핵심 전환점.
                         CRL_SNAPSHOT 노드로 보존.

  Valley of Death        TRL ≥ 5 AND CRL ≤ 3 상황. 기술은 검증됐으나 고객·시장 연결이 없는 사업화
                         위험 구간. L4 Moving Target Gate 즉시 발동 조건. CRL 가속 전략 수립 필수.

  TRL-CRL 갭 (Gap)       TRL 수준과 CRL 수준의 차이. 갭 ≥ 2이면 Valley of Death 위험 증가. 갭 =
                         CRL - TRL ≥ 3이면 사업화 압박 과잉 경고(WARNING).

  CRL_SNAPSHOT           CRL 달성 시 생성되는 Neo4j Append-only 노드. 달성 단계·근거·확인자·서명
                         일시 보존. CRL 5 이상은 PI 서명 포함. 법적·행정적 사업화 근거.

  \[CRL_UP\]             일일보고 태그. 연구원/PM이 CRL 상승 사실을 보고할 때 사용. AI 검증 엔진이
                         달성 근거 첨부 여부 확인. CRL 5 이상은 PM+PI 서명 요청 자동 발동.

  전용 앱 (RAPai PWA)    RAPai의 주 입력 채널. Progressive Web App 방식. 모바일 홈 화면 아이콘.
                         Service Worker 오프라인 지원. Submit → POST /api/reports 직접 호출. 사전
                         AI 차단 가능.

  보조 채널              Outlook·Teams를 통한 선택적 일일보고 채널. 전송 후 사후 AI 피드백 방식.
                         일반·주의 과제만 권장. 보호 과제 사용 제한.

  Webhook Router         Outlook·Teams Webhook을 수신하고 일일보고 여부를 판단하여 메시지 큐로
                         전달하는 RAPai 서버 컴포넌트.

  user_registry          조직 이메일·Teams ID와 RAPai user_id를 매핑하는 테이블. Active Directory와
                         동기화.

  4계층 파이프라인       일일보고 데이터 흐름. Layer 1 입력채널 → Layer 2 수신정규화 → Layer 3
                         AI검증+파싱 → Layer 4 동시이중분기.

  Output KPI             프로젝트가 직접 생산한 산출물 지표. 논문·특허·시제품·보고서. 과제 수행 중
                         측정.

  Outcome KPI            산출물이 만들어낸 변화 지표. 기술이전·사업화·정책 채택. 과제 종료 후
                         1\~3년 측정.

  Value KPI              Outcome이 조직·사회에 가져온 궁극적 편익 지표. 매출·고용·사회 편익. 종료
                         후 3\~5년 측정.

  AG-TEAM                내부 팀 관리 전담 에이전트(12번째). 역량·투입률·과부하·Retrospective·이탈
                         위험 감지.

  AG-QA                  품질보증·오작동 모니터링 에이전트(13번째). 11개 도메인 에이전트
                         감시·드리프트 감지·Human 보고.

  TAILORING_DECISION     착수 시 project_type·lifecycle_template 선택 근거를 Append-only 기록하는
                         MindVault 노드.

  OUTCOME_SNAPSHOT       추적평가 시점(1·3·5년 후)별 Outcome·Value KPI 달성 현황 Append-only 기록
                         노드.

  LEARNING_CYCLE         자가 학습 사이클 기록 노드. before/after 파라미터·증거 건수·승인자·롤백
                         기준값.

  OVERRIDE_DECISION      Human Override 이력 Append-only 기록 노드. 자가 학습 데이터로 환류.

  VUCA 프로파일          Volatility·Uncertainty·Complexity·Ambiguity 4차원 GraphState 필드.

  complexity_level       프로젝트 복잡성 수준. \"low\"·\"medium\"·\"high\"·\"very_high\". Gate
                         엄격도·AG-SENSE 빈도 자동 조정.

  strategic_goal_id      과제가 기여하는 기관 전략 목표 ID. L2 Gate Could 기준 연동.

  Human Oversight        AG-QA 보고를 수신하고 오작동 정정·임계값 변경·자가 학습 적용을 최종
  Manager                승인하는 관리담당자.

  Progressive Disclosure 착수 시 필수 3개(project_type·주요 납기·TRL 목표)만 입력하고 나머지는
  UX                     시스템 기본값으로 순차 확장하는 UX 패턴.

  L1 TRL Gate            TRL 달성을 PI가 근거와 함께 서명·확인하는 기술적 사실 확인 Gate. Go/Kill
                         없음. 유형1·2 전용.

  L2 Stage Gate          Stage 완료 후 기술·시장·사업성을 종합 평가하여 투자
                         지속(Go)/중단(Kill)/보류(Hold)/재작업(Recycle)/조건부(Conditional Go)를
                         결정하는 Gate.

  L3 Annual Gate         정부R&D 과제의 연 1회 또는 단계 완료 시 외부 전문기관·평가위원회가
                         성과목표 달성도를 심의하는 Gate. 국가연구개발혁신법 기반.

  L4 Moving Target Gate  AG-SENSE가 CRITICAL 외부 환경 변동을 감지할 때 자동 발동되는 비정기 유효성
                         검토 Gate. AI 추천, PI 결정, 72h SLA.

  Must 기준              L2 Stage Gate에서 미달 시 즉시 Kill을 권고하는 필수 기준. 기술 달성·예산
                         준수·법적 문제 없음 등.

  Should 기준            L2 Stage Gate에서 점수화하여 가중 합산하는 전략적 가치 기준.
                         시장성·차별성·IP 강화 가능성 등.

  GATE_DECISION          모든 Gate 계층의 결정을 Append-only로 기록하는 MindVault 노드.
                         gate_layer·결정 유형·근거·PI 서명 포함. 삭제·수정 불가.

  TRL Gate vs Stage Gate TRL Gate(L1): 기술적 사실 확인. Stage Gate(L2): 경영적 투자 판단. TRL
                         달성(L1 통과)이 Stage Gate(L2) Must 기준 중 하나이나, L1 통과 ≠ L2 통과.

  gate_layer             GraphState의 현재 활성 Gate 계층 식별자.
                         \"L1_trl\"\|\"L2_stage\"\|\"L3_annual\"\|\"L4_moving_target\"\|\"none\".

  Valley of Death        trl_current≥5 AND crl_current≤3인 상태. L4 Gate 즉시 발동. 유형1·2 전용
                         경보.
