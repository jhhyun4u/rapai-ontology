## 1. 특허 포트폴리오 전체 구조

### 1.1 통합 구조 설계 원칙

v2.0에서는 다음 세 가지 설계 원칙을 적용한다.

**원칙 1 — 핵심 기능은 독립 특허로 분리**
CP-7과 CP-4는 각각 독립적으로 작동하고 독립적인 시장 가치를 가지므로
별도 특허로 출원한다. 하나가 무효화되어도 나머지는 유효하다.

**원칙 2 — 연결 기술은 종속항으로 통합**
CP-1(해시체인 무결성)과 CP-10(IP 신호 감지)은 독립 특허로는
신규성·진보성 입증이 어렵고, CP-7의 맥락 안에서만 완전한 의미를 가진다.
CP-7의 종속항으로 흡수하여 권리 범위를 확대한다.

**원칙 3 — 양방향 피드백 루프를 청구항에 명시**
CP-4(WBS 생성·변경)와 CP-7(에이전트 연쇄 처리)은 단방향 의존이 아닌
양방향 피드백 루프로 작동한다. 이 루프 구조 자체를
종속항에 명시하여 두 특허의 결합을 별도로 보호한다.

## 2. 발명의 배경 및 기술적 과제

### 2.1 R&D 행정 부담의 구조적 문제

연구개발 현장에서 연구자가 실험을 완료하면 다음의 작업들이 요구된다.

- 연구노트에 실험 결과 기록
- 진도 관리 시스템에 Task 완료 여부 수동 입력
- 예산 집행 실적 시스템에 소요 시간·재료비 별도 입력
- 주간 보고서 작성 (동일 내용 재작성)
- Gate review 자료 준비 (동일 데이터 재집계)
- 위험 관리 대장 수동 갱신

**하나의 실험 결과가 최소 5~6개 시스템에 수동으로 중복 입력된다.**
이는 연구자의 핵심 행정 부담으로, 연구 본연의 업무 시간을 잠식하는
구조적 비효율의 원인이다.

### 2.2 R&D 프로젝트 관리의 고유한 어려움

R&D 프로젝트는 일반 건설·제조 프로젝트와 근본적으로 다르다.

**비선형성:** 실험 결과에 따라 계획이 반복적으로 수정된다.
초기에 수립된 WBS가 프로젝트 종료까지 유지되는 경우는 드물다.

**불확실성 내재:** 연구 결과가 예측 불가하여 단순한 일정 관리
도구로는 R&D의 실제 진행 상황을 반영하기 어렵다.

**다중 성과 기준:** 논문·특허·기술이전·사업화 등 다양한 성과
유형이 혼재하며, 단일 지표로 진도를 측정할 수 없다.

**법적 기록 의무:** 국가R&D 과제는 연구노트의 법적 무결성이
요구되며, 데이터 위변조 방지가 필수적이다.

### 2.3 해결하려는 기술적 과제

본 발명군이 해결하고자 하는 기술적 과제는 다음과 같다.

1. **단일 입력의 다중 반영:** 연구자가 기록 한 번으로 모든
   R&D 관리 영역이 자동 갱신되도록 하는 것
2. **WBS의 동적 적응:** 연구 수행 중 발생하는 변경 이벤트를
   자동으로 감지하고 WBS·일정·자원을 재계획하는 것
3. **에이전트 판단의 신뢰성:** AI 에이전트가 판단 근거로
   사용하는 연구 기록의 위변조 불가성을 보장하는 것
4. **IP 신호의 실시간 감지:** 연구 수행 중 발생하는 특허
   가능성 신호를 자동으로 포착하는 것
5. **양방향 루프:** 계획(WBS)과 실행(에이전트)이 서로를
   실시간으로 갱신하는 피드백 구조를 구현하는 것

## 3. 기존 기술의 한계

### 3.1 범용 프로젝트 관리 도구

MS Project, Jira, Asana 등 범용 PM 도구는 각 기능이 독립적으로
운영되며 자동 연결 메커니즘이 없다.

- 연구 기록 입력이 진도·예산·위험 관리와 자동 연결되지 않음
- WBS 변경 시 영향 범위를 수동으로 재계산해야 함
- R&D의 비선형적 특성을 반영한 Stage-Gate 구조 미지원

### 3.2 전자연구노트(ELN) 시스템

LabArchives, Benchling, SciNote 등은 연구 기록의 보존에 특화되어 있으나
PM 기능과의 연동이 없다.

- 기록 저장이 PM 도메인을 자동 갱신하지 않음
- WBS와의 연결 구조 없음
- 기록 데이터 기반 IP 신호 감지 기능 없음

### 3.3 기존 AI 에이전트 시스템

범용 멀티에이전트 시스템은 일반 업무 자동화가 대상이며
R&D 관리의 특수성을 반영하지 못한다.

- 연구 기록 이벤트 기반 자동 트리거 구조 없음
- WBS 변경과 에이전트 처리의 양방향 연동 없음
- 연구 기록 해시 무결성 검증 연동 없음

### 3.4 기존 IP 관리 플랫폼

PatSnap, Derwent Innovation, DeepIP 등은 독립된 IP 분석 플랫폼이다.

- R&D PM 시스템 내장형 실시간 신호 감지 구조 없음
- 연구 기록 저장 이벤트와 연동된 자동 감지 없음
- 에이전트 처리의 일부로서 PM 흐름에 통합되지 않음

## 4. CP-7: 핵심 독립 특허

### 발명의 명칭

**연구 기록 저장 이벤트 기반 다중 에이전트 병렬 연쇄 처리를 통한
연구개발 프로젝트 자동 관리 시스템 및 방법**

### 4.1 핵심 구성 요소

CP-7은 다음 5개 구성 요소로 이루어진다.

```
┌─────────────────────────────────────────────────────┐
│                  연구자 기록 행위                     │
│      (자연어 입력 / 구조화 입력 / 음성 입력)          │
└──────────────────────┬──────────────────────────────┘
                       │ 저장 완료
                       ▼
┌─────────────────────────────────────────────────────┐
│  [구성 1] 이벤트 수신부                              │
│  저장 완료 감지 → 단일 이벤트 객체 생성              │
│  {record_id, task_id, researcher_id, timestamp}     │
└──────────────────────┬──────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────┐
│  [구성 2] 오케스트레이터                             │
│  단일 이벤트 → 복수 에이전트 병렬 호출               │

└──┬──────┬──────┬──────┬──────┬───────┬─────────────┘
   │      │      │      │      │       │
   ▼      ▼      ▼      ▼      ▼       ▼
┌─────────────────────────────────────────────────────┐
│  [구성 3] 전문 에이전트 집합 (병렬 실행)             │
│                                                     │
│  A1 진도관리    A2 마일스톤    A3 위험관리            │
│  Task 완료 판정  Gate 달성도   3채널 위험 탐지        │
│  EVM 지표 갱신  Go/Kill 평가  대응 시나리오 제시      │
│                                                     │
│  A4 성과관리    A5 커뮤니케이션  A6 예산자원          │
│  KPI 측정       보고서 초안    인건비 자동 계상       │
│  IP 신호 감지   알람 발송      번 레이트 분석         │
└──────────────────────┬──────────────────────────────┘
                       │ 읽기/쓰기
                       ▼
┌─────────────────────────────────────────────────────┐
│  [구성 4] 공유 R&D 데이터 레이어 (단일 진실 소스)    │
│  해시 무결성 검증 데이터만 판단 근거로 허용           │
│  모든 에이전트 처리 결과 통합 보관                   │
└──────────────────────┬──────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────┐
│  [구성 5] 완료 피드백부                              │
│  도메인별 변화 내역을 연구자에게 즉시 표시            │
│  "무엇이 바뀌었는가"를 구체적으로 제공               │
└─────────────────────────────────────────────────────┘
```

### 4.2 에이전트별 처리 내용

| 에이전트 | 핵심 처리 | 판단 방식 | WBS 변경 연동 |
|---|---|---|---|
| A1 진도관리 | Task DoD 충족 여부, EVM 산출 | 규칙 엔진 (AI 추론 배제) | 변경된 DoD 즉시 반영 |
| A2 마일스톤 | Gate 달성도, 등급 부여 | DB 직접 집계 | 변경된 Gate 기준 반영 |
| A3 위험관리 | 3채널 탐지, 스코어링 | 규칙·패턴·AI 병렬 | 변경 영향 위험 재평가 |
| A4 성과관리 | KPI 갱신, IP 신호 감지 | RAG 기반 + 규칙 | 변경 후 성과 재산출 |
| A5 커뮤니케이션 | 보고서 초안, 알람 분류 | 템플릿 + 데이터 채움 | 변경 사실 이해관계자 통보 |
| A6 예산자원 | 소요시간 × 단가, 번 레이트 | 규칙 엔진 | 변경된 자원 계획 반영 |

### 4.3 실제 처리 시나리오

```
[이벤트 발생]
연구자 김OO, 실험 결과 기록 저장
"촉매 0.5%, 수율 87.3%, 소요 6.5시간"

[오케스트레이터: A1~A6 병렬 호출, 총 3초 이내]

A1 진도관리:
  → Task S2-A3-T1 DoD 충족 (수율 85% 이상) 판정
  → Stage 2 진도 78% → 83% 갱신
  → SPI 0.94 (정상)

A2 마일스톤:
  → Gate 2 재현성 3/3회 달성 확인
  → YELLOW(72%) → GREEN(88%) 전환
  → Gate 2 증빙 패키지 자동 업데이트

A3 위험관리:
  → "수율 미달" 위험 해소 조건 충족
  → 위험 레지스터 자동 종료

A4 성과관리:
  → 수율 87.3% → IP 임계값(85%) 초과 감지
  → 특허 체크리스트 자동 생성
  → 유사 선행특허 3건 자동 검색

A5 커뮤니케이션:
  → PM에게 Gate 2 GREEN 달성 알람 발송
  → 일일 보고서 초안 갱신

A6 예산자원:
  → 6.5시간 × 인건비 단가 자동 계상
  → 예산 소진율 63.1% 갱신

[완료 피드백]
연구자 화면에 즉시 표시:
  ✓ Task 완료 · Stage 2 진도 83%
```

### 5.3 WBS 변경 이벤트의 세 가지 유형

**유형 1 — 실험 결과 기반 변경**

```
  5. Gate 2 일정 자동 조정
  6. CP-7 A2 에이전트에 Gate 준비도 재산출 신호 발송
```

**유형 2 — 신규 발견 기반 변경**

```

  1. 변경 유형: Stage 구조 재편 (관리자 승인 필요)
  2. 승인 요청 → 관리자 검토 → 승인
  3. 신규 Stage 삽입 및 Gate Criteria 재정의
  4. 전체 임계 경로 재계산
  5. 자원 재배분 시뮬레이션
  6. CP-7 전체 에이전트에 갱신 이벤트 일괄 발송
```

**유형 3 — 자원 변동 기반 변경**

CP-7의 A6 예산자원 에이전트가 자원 변동을 감지하면 변경 엔진이 작동한다.

```

변경 엔진 처리:
  1. 변경 유형: 자원 변동 기반
  2. 영향 Task 식별: 해당 연구원 배정 Task 전체
  3. 일정 재산출: 가용 자원 기준 완료 가능 시점 재계산
  4. 3가지 시나리오 제시:
     - 시나리오 A: 현행 유지 → Gate 45일 지연
     - 시나리오 B: 외부 협력 투입 → Gate 15일 지연
```

## 6. CP-1: 종속항 통합 (에이전트 판단 신뢰 기반)

### 6.1 재정의 — R&D 관리와의 연결

CP-1은 단순한 "해시체인 기록 봉인 기술"이 아니다.
**CP-7 에이전트들이 내리는 모든 판단의 신뢰 기반**이다.

CP-7의 진도관리 에이전트(A1)가 "수율 87.3% → Task 완료"를 판정할 때,
그 87.3%라는 수치가 사후에 조작되지 않았음이 보장되어야 한다.
이 보장을 제공하는 것이 CP-1이다.

```
CP-1이 없을 때의 문제:
  연구자 기록: 수율 71.2% (DoD 미달)
  [사후 수정 가능성 존재]
  AI 에이전트 판단 근거: 수율 87.3% (조작된 값?)
  → Gate 2 GREEN 판정 → 3억 원 집행
  → 감사에서 조작 의혹 → 연구부정 혐의

CP-1이 있을 때:
  연구자 기록: 수율 71.2% → 해시 H1 생성·봉인
  [수정 시 해시 불일치 즉시 감지]
  AI 에이전트 판단 근거: 해시 검증된 값만 허용
  → 신뢰할 수 있는 판정 → 법적으로 유효한 의사결정
```

### 6.2 해시체인 구조

```
기록 저장 시:
{
  "entry_id": "REC-20260407-007",
  "prev_hash": "sha256:a3f9b2...",    ← 이전 기록의 해시
  "timestamp": "2026-04-07T14:32:00",
  "researcher_id": "R-042",
  "task_id": "WBS-S2-A3-T1",
  "data": { "yield": 87.3, "purity": 99.1 },
  "this_hash": "sha256:c7d1e4..."     ← 이 기록 전체의 해시
}

무결성 검증 원리:
  기록을 수정하면 this_hash가 변경됨
  → 다음 기록의 prev_hash와 불일치
  → 체인 전체가 파손됨
  → 조작 사실이 수학적으로 증명됨
```

### 6.3 CP-7 종속항 B로의 통합

CP-1의 핵심 내용을 CP-7 종속항 B에 다음과 같이 포함한다.

> **[CP-7 종속항 B]** 제1항에 있어서,
> 상기 공유 R&D 데이터 레이어에 저장된 연구 기록은
> 직전 기록의 해시값을 포함하는 해시체인으로 봉인되어 있으며,
> 상기 복수의 전문 에이전트는 해시체인 무결성이 검증된
> 연구 기록 데이터만을 판단 근거로 사용하고,
> 무결성 검증 실패 시 해당 데이터를 판단에서 제외하며
> 관리자에게 경보를 발생시키는 것을 특징으로 하는 시스템.

**CP-1을 종속항으로 통합하는 이유:**
독립 특허로 출원 시 "해시체인 봉인"은 선행기술이 다수 존재하여
신규성 주장이 약하다. 반면 "R&D 에이전트 판단 신뢰 기반으로서의
해시체인"은 이 맥락에서만 성립하는 특화된 적용으로,
CP-7의 종속항 내에서 더 강한 보호가 가능하다.

## 7. CP-10: 종속항 통합 (IP 신호 자동 감지)

### 7.1 재정의 — R&D 관리 흐름 내 통합

CP-10을 독립 특허로 두면 두 가지 약점이 발생한다.

**약점 2 — 추상성 리스크:** "실험 결과에서 IP 신호를 감지한다"는
행위 자체는 사람도 할 수 있어 특허 심사관이 추상적 아이디어로
거절할 가능성이 있다.

CP-7의 A4 성과관리 에이전트 내부 기능으로 포함하면 상황이 달라진다.
**"연구 기록 저장 이벤트 발생 시 성과관리 에이전트가 자동으로
결과값을 IP 임계값과 대조하여 특허 체크리스트를 생성한다"**는 것은
R&D PM 흐름 안에서만 의미가 있는 특화된 자동화 구조이다.

### 7.2 CP-7 종속항 C로의 통합

> **[CP-7 종속항 C]** 제1항에 있어서,
> 상기 성과관리 에이전트는 상기 연구 기록 데이터의 결과값을
> 기술 분야별로 사전 정의된 지식재산 신호 임계값과 자동 대조하고,
> 임계값 초과 또는 신규 조건 조합 발견 시 외부 특허 데이터베이스를
> 실시간 조회하여 유사 선행특허 검색 결과 및 특허 출원 체크리스트를
> 자동 생성하며, 발명 공개 방지 알람을 연구자에게 즉시 발송하는
> 것을 특징으로 하는 시스템.

### 8.3 CP-7 종속 청구항 전체

**[종속항 A — CP-4 DoD 연결]**
제1항에 있어서, 상기 공유 R&D 데이터 레이어에 저장된
Task 완료 기준(Definition of Done)은 연구계획서를 파싱하여
산업 유형·연구 단계·기술성숙도(TRL) 수준의 3차원 분류에 기반해
자동 생성된 것이며, 상기 진도관리 에이전트는 AI 추론 모듈을 배제하고
규칙 엔진을 이용하여 상기 Task 완료 기준의 충족 여부를 판정하는
것을 특징으로 하는 시스템.

**[종속항 B — CP-1 해시 무결성]**
제1항에 있어서, 상기 공유 R&D 데이터 레이어에 저장된 연구 기록은
직전 기록의 해시값을 포함하는 해시체인으로 봉인되어 있으며,
상기 복수의 전문 에이전트는 해시체인 무결성이 검증된 연구 기록
데이터만을 판단 근거로 사용하고, 무결성 검증 실패 시 해당 데이터를
판단에서 제외하며 관리자에게 경보를 발생시키는 것을 특징으로 하는 시스템.

**[종속항 C — CP-10 IP 신호 감지]**
제1항에 있어서, 상기 성과관리 에이전트는 상기 연구 기록 데이터의
결과값을 기술 분야별로 사전 정의된 지식재산 신호 임계값과 자동 대조하고,
임계값 초과 또는 신규 조건 조합 발견 시 외부 특허 데이터베이스를
실시간 조회하여 유사 선행특허 검색 결과 및 특허 출원 체크리스트를
자동 생성하며, 발명 공개 방지 알람을 연구자에게 즉시 발송하는
것을 특징으로 하는 시스템.

**[종속항 D — Gate 증빙 패키지]**
제1항에 있어서, 상기 마일스톤 에이전트는 사전 정의된 Gate 기준의
달성 여부를 상기 공유 데이터 레이어에서 직접 집계하여 산출하고,
달성도에 따라 GREEN·YELLOW·RED 등급을 부여하며, 달성도가 설정
임계값 이상인 경우 달성 실험 기록, 미달 항목, 근거 데이터를 포함한
Go/Kill 증빙 패키지를 자동 생성하는 것을 특징으로 하는 시스템.

**[종속항 E — 예산 자동 계상]**
제1항에 있어서, 상기 예산자원 에이전트는 상기 연구 기록 데이터에
포함된 작업 소요 시간 및 사용 자원 정보에 사전 정의된 단가를 적용하여
연구비 집행 실적을 자동 산출하고, 현재 집행 속도에 기반한
잔여 예산 시뮬레이션을 수행하는 것을 특징으로 하는 시스템.

**[종속항 F — 장애 격리 재시도]**
제1항에 있어서, 상기 오케스트레이터는 특정 전문 에이전트의 처리
실패 시 해당 에이전트만을 재시도 큐에 등록하고 나머지 에이전트의
처리 완료 상태를 유지하는 것을 특징으로 하는 시스템.

**[종속항 G — WBS 변경 연동]**
제1항에 있어서, 상기 진도관리 에이전트 또는 예산자원 에이전트가
Task 완료 기준 미충족, 반복 실험 필요, 또는 자원 변동을 감지한 경우,
WBS 동적 변경 엔진에 변경 이벤트를 발생시키고,
상기 WBS 동적 변경 엔진이 영향 범위 산출 및 재계획을 완료한 후
변경된 WBS 기준을 상기 공유 데이터 레이어에 반영하면,
상기 오케스트레이터가 관련 에이전트를 재호출하여 갱신된 기준으로
관리 도메인을 재계산하는 것을 특징으로 하는 시스템.

**[컴퓨터 판독 가능 매체 청구항]**
컴퓨터로 실행 가능한 명령어를 저장한 비일시적 컴퓨터 판독 가능 매체로서,
상기 명령어는 프로세서에 의해 실행될 때 제2항 내지 제8항 중
어느 한 항의 방법을 수행하는 비일시적 컴퓨터 판독 가능 매체.

### 8.4 CP-4 독립 청구항

**CP-4 독립 청구항 1 — 초기 WBS 자동 생성**

연구개발 프로젝트 구조 자동 생성 방법으로서,

(a) 연구계획서 문서에서 NLP를 이용하여 기술 분야, 연구 목적,
    예산, 기간, 목표 성과를 추출하는 단계;

(b) 추출된 정보로부터 산업 유형, 연구 단계, 기술성숙도(TRL)
    목표 수준을 각각 분류하는 3차원 분류 단계;

(c) 분류 결과의 조합에 대응하는 Stage-Gate 구조 템플릿을
    선택하는 단계; 및

(d) 선택된 템플릿을 기반으로 각 Stage의 Activity와 Task를
    계층적으로 분해하고, 각 Task에 대해 측정 가능한
    완료 기준(Definition of Done)을 자동 생성하는 단계;

를 포함하는 R&D 프로젝트 구조 자동 생성 방법.

**CP-4 독립 청구항 2 — 동적 WBS 변경 엔진**

연구개발 프로젝트 수행 중 WBS 동적 변경 방법으로서,

(a) 연구 수행 과정에서 발생하는 변경 이벤트를
    실험결과 기반·신규발견 기반·자원변동 기반으로 분류하는 단계;

(b) 변경 유형에 따라 영향을 받는 Task·Activity·Stage의
    범위를 자동 산출하는 단계;

(c) 영향 범위 내의 WBS 구조, 일정, 자원 배분을 자동 재계산하는 단계;

(d) 변경 전·후의 임계 경로(Critical Path) 변화를 비교하여
    전체 종료 예정일의 영향을 산출하는 단계;

(e) WBS 변경 이력을 버전으로 관리하고 변경 사유와 함께 보존하는 단계; 및

(f) 변경 완료 시 관련 AI 에이전트에 WBS 갱신 이벤트를
    발생시키는 단계;

를 포함하는 R&D WBS 동적 변경 방법.

**CP-4 종속 청구항 — CP-7 양방향 연동**

제2항에 있어서, 상기 (a) 단계의 변경 이벤트는 연구 기록 저장
이벤트에 응답한 AI 에이전트로부터 수신되고, 상기 (f) 단계의
WBS 갱신 이벤트에 응답하여 진도관리 에이전트, 마일스톤 에이전트,
위험관리 에이전트, 예산자원 에이전트가 변경된 WBS 기준으로
각 관리 도메인을 자동 재계산하는 것을 특징으로 하는 방법.

## 10. 길목 특허 전략 분석

### 10.1 CP-7의 길목 특허 요건 충족 여부

경쟁자가 CP-7을 침해하지 않으면서 동일 서비스를 만들 수 있는가?

**우회 옵션 A: 이벤트 트리거 대신 수동 트리거 사용**
→ 핵심 가치 명제("기록 한 번으로 모두 처리") 포기
→ 시장 경쟁력 근본 상실

**우회 옵션 B: 순차 처리 방식으로 변경**
→ 처리 시간 병렬(3초) 대비 순차(18초+) 급증
→ 사용자 경험 저하로 실용성 상실

**우회 옵션 C: 에이전트 수 축소**
→ R&D 관리 자동화 완결성 상실
→ "복수의 전문 에이전트"로 광범위하게 청구되어 있어 우회 어려움

**결론: 세 가지 우회 옵션 모두 서비스 핵심 가치를 훼손.**
CP-7은 진정한 길목 특허 요건을 충족한다.

### 10.2 CP-4의 길목 특허 요건 충족 여부

**우회 옵션 A: 초기 WBS만 생성, 변경은 수동으로**
→ 연구 수행 중 계획-실제 괴리 문제 미해결
→ R&D 특성을 반영하지 못하는 시스템으로 시장에서 외면

**우회 옵션 B: 변경 처리하되 에이전트 연동 없이**
→ 변경 사실이 에이전트에 반영되지 않아 잘못된 판정 지속
→ 시스템 신뢰성 붕괴

**결론: CP-4의 동적 변경 엔진도 R&D PM 시스템에서
우회 불가능한 길목 기술이다.**

### 10.3 CP-7 + CP-4 결합의 이중 장벽 효과

두 특허를 동시에 보유하면 다음의 이중 장벽이 형성된다.

```
경쟁사가 유사 서비스를 출시하려면:

CP-7 장벽: 이벤트 기반 에이전트 연쇄 처리를 우회해야 함
           → 우회 시 핵심 자동화 가치 상실

CP-4 장벽: 동적 WBS 변경 엔진을 우회해야 함
           → 우회 시 R&D 특화 계획 관리 불가

두 장벽을 동시에 우회하면서 경쟁력 있는 서비스 유지 = 사실상 불가능
→ 라이선스 협상 또는 시장 포기 선택
```

| 용어 | 정의 |
|---|---|

| CPM | Critical Path Method — 프로젝트 최장 경로 분석 기법 |
| RAG | Retrieval-Augmented Generation — 외부 DB 검색 기반 AI 생성 방식 |
| 해시체인 | 각 기록에 이전 기록의 해시값을 포함시켜 연결하는 위변조 방지 구조 |
